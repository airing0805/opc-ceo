## 版本规范

### 命名规则
- 格式：`v{n}-名称`
- 示例：`v3-战略落地`、`v4-系统优化`

### 版本周期
- **理想周期**：1-2 天一个版本
- **升级时机**：当前版本所有任务完成后立即升级
- **版本封存**：完成后不再改动，只做记录

### 聚焦原则
- 只做当前版本的事
- 旧版本封存，不再改动
- 新需求放入规划池（见下文）

## 规划池

> 优先级较低的待规划事项

### 规则

| 字段 | 说明 |
|------|------|
| 优先级 | P0（紧急）/ P1（高）/ P2（中）/ P3（低） |
| 状态 | 待规划 / 进行中 / 已提取 |
| 来源 | 记录这个 idea 从哪里来 |

**P1 - 高优先级**

|----|------|------|------|

**P3 - 低优先级**

| ID | 事项 | 来源 | 状态 |

| P3-03 | 自动化能力增强 | 公司愿景.md | 待规划 |

### 更新规则

1. **新 idea**：随时添加到规划池，标注优先级
2. **提取执行**：当前版本开始时，从规划池提取 P1/P2 事项
3. **版本升级**：升级版本时，清空已提取事项的状态
4. **定期清理**：每季度清理一次长期未执行的 P3 事项
