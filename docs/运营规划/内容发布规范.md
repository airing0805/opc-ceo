## 三、各平台快速参考

### CSDN 发布规范摘要

**标题格式**：
```
# 一文讲清 XXX（2026最火XXX）
# XXX 是什么？一篇文章讲清楚
```

**内容结构**：
1. **前言**：一句话抓住注意力，说明本文解决什么问题
2. **比喻/类比**：用通俗的比喻讲清楚核心概念
3. **为什么重要**：讲清楚为什么读者需要关注
4. **怎么用**：实操步骤，代码+命令
5. **适合谁**：明确目标受众
6. **总结**：强化记忆
7. **下期预告**：引导关注

**标签规范**：
```
MCP, Claude Code, AI编程, 2026, 入门教程
```

> 完整规范见：[CSDN发布指南](平台发布指南/CSDN发布指南.md)

### 简书 发布规范摘要

**代码块规范**：
- 必须标注语言，支持语法高亮
- 单个代码块不超过50行
- 代码可运行，包含必要依赖

**专题投稿**：
- 投稿到相关专题，增加曝光
- 选择关注人数多的专题
- 一篇文章最多投5个专题

## 四、内容生产流程

```
选题库 → 写初稿 → 平台适配 → 发布 → 数据跟踪
```

### 核心原则

1. **一次只写一篇**：不贪多
2. **模板化**：固定结构，只填变量
3. **少迭代**：初稿即终稿，微调就发
4. **实测为王**：只写自己用过的

## 七、发布前验证检查清单

> 为解决"草稿箱→直接发布，无质量验证"的问题，建立发布前验证机制，确保内容质量。

### 1. 检查清单结构

| 检查项 | 检查内容 | 权重 | 检查方法 |
|--------|----------|------|----------|
| AI味检测 | 过渡词≤5个、无套路开头结尾 | 30% | 自动化检测 |
| 内容完整性 | 案例≥2个、代码示例≥1个 | 文本分析 |
| 格式规范 | 标题层级、代码块、配图 | 20% | 结构分析 |
| 可读性 | 段落长度、阅读流畅度 | 15% | 难度评估 |
| 重复度 | 原创度<80% | 10% | 相似度检测 |

### 2. 每项检查的具体标准

#### 2.1 AI味检测（权重：30%）

**检查目标**：识别AI套路化表达，降低"AI味"

**检查标准**：

| 维度 | 标准 | 阈值 |
|------|------|------|
| 过渡词 | 禁止使用AI高频过渡词 | ≤5个/篇 |
| 开头格式 | 禁止"随着"、"在当今"、"近年来"等套路开头 | 0个 |
| 结尾格式 | 禁止"总之"、"以上就是"、"希望本文能帮助"等套路结尾 | 0个 |
| 绝对化用语 | 禁止"最XX"、"绝对"、"100%"等过度营销用语 | ≤1个 |

**AI高频过渡词黑名单**：

```
首先、其次、最后、此外、另外、与此同时、值得注意的是、
需要注意的是、综上所述、由此可见、不言而喻、显而易见、
不难发现、由此可见、事实上、实际上、从某种程度上来说
```

**判定方法**：
- 统计文章中出现的黑名单词汇数量
- 检查开头50字和结尾50字是否包含套路表达
- 阈值判定：过渡词>5个 或 存在套路开头/结尾 → 不通过

---

#### 2.2 内容完整性（权重：25%）

**检查目标**：确保内容有足够的信息量和实践价值

**检查标准**：

| 维度 | 标准 | 阈值 |
|------|------|------|
| 案例数量 | 必须包含真实案例 | ≥2个 |
| 代码示例 | 必须包含可运行代码 | ≥1个 |
| 步骤说明 | 必须有可操作的步骤 | ≥3步 |
| 总结要点 | 必须有明确的总结 | 1段 |

**判定方法**：
- 案例检测：查找"例如"、"比如"、"以XXX为例"、"案例："等关键词
- 代码检测：查找代码块标记（```或```python/```javascript等）
- 步骤检测：查找有序列表（1. 2. 3.）或"第一步"、"第二步"等
- 阈值判定：任一维度不达标 → 不通过

---

#### 2.3 格式规范（权重：20%）

**检查目标**：确保内容符合平台展示规范

**检查标准**：

| 维度 | 标准 | 阈值 |
|------|------|------|
| 标题层级 | 必须有二级以上标题 | H1+H2≥3个 |
| 代码块 | 必须有带语言标注的代码块 | ≥1个 |
| 段落长度 | 单段落不超过150字 | 平均≤150字 |
| 列表使用 | 列举内容使用列表形式 | 适当使用 |

**判定方法**：
- 标题检测：统计#和##出现次数
- 代码块检测：检查是否有带语言标识的代码块
- 段落检测：计算平均段落长度
- 阈值判定：标题<3个 或 无代码块 → 不通过

---

#### 2.4 可读性（权重：15%）

**检查目标**：确保内容易于阅读和理解

**检查标准**：

| 维度 | 标准 | 阈值 |
|------|------|------|
| 段落长度 | 单段落不超过150字 | 最长段落≤150字 |
| 句子长度 | 单句不超过40字 | 平均≤40字/句 |
| 专业术语 | 需有解释或举例 | 每个术语有解释 |
| 阅读难度 | 避免过于专业的术语堆砌 | 句子结构不过于复杂 |

**判定方法**：
- 段落分析：拆分段落，统计每段字数
- 句子分析：按句号拆分，统计每句长度
- 术语检测：识别可能需要解释的术语
- 阈值判定：最长段落>150字 → 建议优化

---

#### 2.5 重复度检测（权重：10%）

**检查目标**：确保内容原创度，避免重复内容

**检查标准**：

| 维度 | 标准 | 阈值 |
|------|------|------|
| 原创度 | 与网络现有内容相似度 | <80%相似 |
| 重复段落 | 连续重复字数 | <100字连续重复 |

**判定方法**：
- 选取文章关键段落（开头、核心观点、结论）
- 与网络搜索结果对比
- 检测连续重复字符数
- 阈值判定：原创度<80% → 不通过

---

### 3. 综合判定

**通过条件**：所有检查项均通过（或可接受范围）

| 结果 | 条件 | 行动 |
|------|------|------|
| 通过 | 所有检查项通过 | 可直接发布 |
| 需优化 | 1-2项不通过（AI味/可读性） | 优化后重新检查 |
| 需重写 | 3项以上不通过 | 返回内容生产环节 |

**权重得分计算**：

```
总分 = AI味得分×0.30 + 完整性得分×0.25 + 格式得分×0.20 + 可读性得分×0.15 + 原创度得分×0.10

通过阈值：总分≥70分
```

### 4. 自动化检测工具

3. **格式检查脚本**：验证标题层级、代码块规范性
4. **原创度检测**：与知识库内容对比

## 八、发布审批流程设计

> 为确保发布内容质量，建立完整的审批流程，从提交到发布经过多层审核。

### 1. 流程节点

| 节点 | 执行者 | 操作 | 产出 |
|------|--------|------|------|
| 1. 提交审核 | 内容生产者 | 提交待发布内容 | 提交记录 |
| 2. 自动检测 | AI检测Agent | 运行AI味检测 | 检测报告 |
| 3. 质量评估 | 质量评估Agent | 综合评分 | 评估结果 |
| 4. 人工复核 | 人工审核 | 最终确认/退回 | 审核决定 |
| 5. 发布执行 | 发布Agent | 正式发布 | 发布记录 |

**流程图**：

```
内容生产者
    │
    ▼
提交审核 ──────────────┐
    │                   │
    ▼                   │
AI检测Agent             │
    │                   │
    ├─检测通过──────────┤
    │                   │
    ▼                   │
质量评估Agent           │
    │                   │
    ├─评估通过──────────┤
    │                   │
    ▼                   │
人工复核               │
    │                   │
    ├─审核通过──────────┤
    │                   │
    ▼                   │
发布Agent              │
    │                   │
    ▼                   │
发布成功 ◄─────────────┘
```

---

### 2. 审批规则

#### 2.1 综合评分标准

| 综合得分 | 评级 | 处理方式 |
|----------|------|----------|
| ≥85分 | A级 | 直接发布 |
| 70-84分 | B级 | 人工复核后发布 |
| 50-69分 | C级 | 退回报错修改 |
| <50分 | D级 | 退回重写 |

#### 2.2 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| AI味检测 | 30% | 过渡词、套路表达、开头结尾 |
| 内容完整性 | 25% | 案例数量、代码示例、步骤说明 |
| 格式规范 | 20% | 标题层级、代码块、配图 |
| 可读性 | 15% | 段落长度、阅读流畅度 |
| 原创度 | 10% | 与网络内容相似度 |

---

### 3. 检测不通过的处理

#### 3.1 过渡词过多

**问题表现**：文章中过渡词超过5个

**处理方式**：
- 标记具体位置（段落/行号）
- 建议替换方案
- 提供无过渡词改写示例

**示例**：
```
第3段第2行："首先" → 建议删除或改为具体衔接
第5段第1行："其次" → 建议改为具体步骤说明
```

#### 3.2 缺少案例

**问题表现**：案例数量少于2个

**处理方式**：
- 建议添加案例的方向（从哪个角度）
- 提供案例模板参考
- 列出可借鉴的已有案例

**示例**：
```
建议添加方向：
1. 真实项目案例：描述一个实际使用场景
2. 个人经验分享：自己的使用心得
3. 对比案例：与其他工具的对比分析
```

#### 3.3 原创度不足

**问题表现**：与网络内容相似度≥80%

**处理方式**：
- 标记重复段落（具体位置）
- 建议改写方向
- 要求补充个人独特观点

**示例**：
```
重复段落：
- 第2-4段：概念定义部分重复度过高
- 第8段：与其他文章观点高度相似

建议：
- 增加个人使用经验
- 添加独特见解和分析
- 补充原创代码示例
```
