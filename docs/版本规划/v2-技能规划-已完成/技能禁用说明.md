# 技能禁用说明

## 概述

本文档说明 OPC-CEO 系统中技能禁用的原因、被禁用技能列表以及恢复计划。

---

## 禁用背景

**禁用时间**：2026-02-24

**禁用原因**：

1. **V2 架构转变** - 从 V1 的 Skill 优先架构转变为 V2 的 Agent 协作架构
2. **简化系统复杂度** - 避免同时维护 Skill 和 Agent 两套体系
3. **聚焦核心能力** - 集中资源开发关键能力（CEO + 教练能力）
4. **响应反馈** - 用户反馈要求简化系统，减少不必要的技能

## 架构转变

### V1 架构（Skill 优先）

```
用户 → CEO (opc-ceo-core skill) → 各角色 Skill
      ↓           ↓                      ↓
  自然语言   MCP Memory          直接技能调用
```

**特点**：
- 基于 Claude Code Skills 系统
- 每个角色是独立的 Skill
- 通过 MCP Memory 共享数据
- 需要定义完整的 Skill 接口

### V2 架构（Agent 协作）

```
用户 → CEO (原生) → Task 工具 → Claude Code Subagents
      ↓           ↓              ↓
  自然语言   TaskList       general-purpose agents
                           (通过 prompt 指导)
```

**特点**：
- 基于原生 Agent 系统
- 使用 Task 工具调用通用 agents
- 通过 prompt 指导 agent 行为
- 无需定义独立 skill 文件

## 恢复计划

### 短期（V2 阶段）

**目标**：验证 V2 架构是否满足需求

| 阶段 | 时间 | 目标 | 验收标准 |
|------|------|------|----------|
| 阶段 1 | V2.1 - V2.4 | 完成架构转变 | 文档沟通规范正常运行 |
| 阶段 2 | 1-2 周 | 收集用户反馈 | 用户满意度提升 |
| 阶段 3 | 2-4 周 | 评估 V2 优缺点 | 明确是否需要回退 |

### 中期（V3 规划）

**可能的恢复场景**：

| 场景 | 触发条件 | 恢复方案 |
|------|----------|----------|
| 场景 1 | V2 架构不满足复杂需求 | 保留 V2 基础，为关键能力恢复 Skill |
| 场景 2 | 通用 agent 精度不足 | 恢复 `task-manager`、`file-manager` 等核心 Skill |
| 场景 3 | 性能或成本问题 | 混合模式：简单用 Agent，复杂用 Skill |

### 长期（V4+ 规划）

**可能的演进方向**：

| 方向 | 说明 |
|------|------|
| 方向 1 | 完全基于 Agent + Prompt，不再维护 Skill |
| 方向 2 | 混合模式，Agent 负责协调，Skill 负责专业能力 |
| 方向 3 | Skill 形式化，通过代码生成而非手动维护 |

## 恢复步骤

如果需要恢复某个技能，按以下步骤操作：

### 步骤 1：评估恢复需求

```markdown
恢复前评估：

1. 为什么需要恢复？
   - [ ] Agent 方案无法满足
   - [ ] 需要更高的准确性
   - [ ] 需要更专业的能力

2. 恢复范围？
   - [ ] 恢复完整 Skill
   - [ ] 恢复部分模块
   - [ ] 重新设计 Skill

3. 兼容性？
   - [ ] 与 V2 架构兼容
   - [ ] 需要修改 V2 架构
   - [ ] 独立运行
```

### 步骤 2：移动技能回 active 目录

```bash
# 从 disabled 移回 skills 目录
mv .claude/skills/disabled/{skill-name} .claude/skills/{skill-name}

# 验证文件完整性
ls -la .claude/skills/{skill-name}/
```

### 步骤 3：更新文档

```markdown
1. 更新 CLAUDE.md 的实现状态
2. 更新 V1 或 V2 规划文档
3. 记录恢复原因到本文档
```

### 步骤 4：测试验证

```markdown
1. 测试 Skill 调用
2. 验证与其他组件的兼容性
3. 更新任务状态
```
