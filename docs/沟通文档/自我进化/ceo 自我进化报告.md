## 二、自我诊断

### 2.1 问题识别

#### 问题1：底层执行能力.md 冗长

**现状**：
- 文档约 1130 行，查找信息困难
- 决策树过于复杂，实际操作效率低
- 错误处理流程过于繁琐

**对比 Coach**：
- Coach 的底层执行能力.md 精简到约 260 行
- 有独立的底层执行示例库.md 存放详细示例
- 核心内容快速索引化

**改进方案**：
```
原底层执行能力.md (1130行)
    │
    ├── 精简核心内容 ──► 新底层执行能力.md (~400行)
    │   ├── 工具选择决策树（简化版）
    │   ├── 常用路径速查
    │   ├── Memory 实体速查
    │   ├── 快速执行模板
    │   └── 错误处理快速决策
    │
    └── 分离详细内容 ──► 底层执行示例库.md (~700行)
        ├── 文档操作详细示例
        ├── Memory 操作详细示例
        ├── 错误处理完整流程
        └── 场景实战示例
```

#### 问题2：WRAP 执行检查不够具体

**现状**：
- WRAP 框架已定义，但执行检查点不够明确
- 缺少强制性的执行前检查
- 思考深度难以量化

**对比 Coach**：
- Coach 有 TRS 框架的强制执行检查点
- 每次回应前必须检查 T/R/S 三个维度
- 有明确的"如果不满足则禁止执行"规则

**改进方案**：
```
WRAP 执行检查点（3秒快速检查）：

每次回应前必须检查:
□ W: 用一句话说清楚用户真正要什么 → 不清楚 → 必须追问
□ R: 指出至少 1 个潜在风险 → 有重大风险 → 必须提示
□ A: 提供至少 2 个方案对比 → 有更好方案 → 必须提供
□ P: 确定至少 1 个需要追问的问题 → 有信息缺口 → 必须追问

如果 W/R/A/P 任一项不满足，禁止给出执行建议！
```

#### 问题3：主动追问模板不够场景化

**现状**：
- 有通用的追问模板，但缺少具体场景适配
- 追问触发条件不够明确

**改进方案**：
增加6大场景的具体追问模板：

| 场景 | 触发词 | 必问问题 | 追问方向 |
|------|--------|----------|----------|
| 咨询场景 | "怎么做XX" | "为什么想解决XX问题？背景是什么？" | 业务价值→影响范围→期望结果 |
| 执行场景 | "去执行XX" | "执行结果有什么具体验收标准？" | 验收标准→时间要求→资源限制 |
| 汇报场景 | "有什么要做的" | "想先看哪类任务？优先级依据是什么？" | 优先级→关注点→下一步计划 |
| 追问场景 | 需求不明 | "能具体描述一下XX是指什么吗？" | 具体定义→边界条件→示例 |
| 错误处理 | 执行失败 | "之前是否正常？最近有什么变化？" | 时间线→变化点→影响范围 |
| 自我进化 | "进化" | "最想改进哪方面能力？" | 改进方向→优先级→验收标准 |

### 3.3 主动追问模板库扩展

**6 大场景追问模板**：

#### 场景1：咨询场景

| 追问点 | 必问问题 | 目的 |
|--------|----------|------|
| 动机 | "为什么想解决XX问题？" | 理解真实需求 |
| 背景 | "这个问题是什么时候开始的？" | 了解上下文 |
| 期望 | "期望达到什么效果？" | 明确成功标准 |
| 范围 | "影响范围有多大？" | 评估影响 |

#### 场景2：执行场景

| 追问点 | 必问问题 | 目的 |
|--------|----------|------|
| 标准 | "执行结果有什么验收标准？" | 明确质量要求 |
| 时间 | "什么时间之前需要完成？" | 了解紧急程度 |
| 资源 | "有什么限制条件？" | 了解约束 |

#### 场景3：汇报场景

| 追问点 | 必问问题 | 目的 |
|--------|----------|------|
| 优先级 | "想先看哪类任务？" | 确定关注点 |
| 依据 | "优先级依据是什么？" | 理解排序逻辑 |
| 下一步 | "接下来想做什么？" | 引导行动 |

#### 场景4：追问场景（需求不明）

| 追问点 | 必问问题 | 目的 |
|--------|----------|------|
| 定义 | "能具体描述一下XX是指什么吗？" | 澄清概念 |
| 边界 | "有什么不包括的情况吗？" | 明确边界 |
| 示例 | "能举个具体例子吗？" | 具体化 |

#### 场景5：错误处理

| 追问点 | 必问问题 | 目的 |
|--------|----------|------|
| 时间线 | "之前是否正常？什么时候开始出问题？" | 定位时间点 |
| 变化 | "最近有什么变化？" | 识别触发因素 |
| 影响 | "影响范围有多大？" | 评估严重程度 |

#### 场景6：自我进化

| 追问点 | 必问问题 | 目的 |
|--------|----------|------|
| 方向 | "最想改进哪方面能力？" | 确定改进重点 |
| 优先级 | "优先改进哪一项？" | 排序改进项 |
| 标准 | "怎么判断改进成功？" | 明确验收标准 |

### 3.4 与 Coach 协作边界优化

**职责矩阵**：

```
┌─────────────────────────────────────────────────────────────┐
│   Coach (ceo-coach)           CEO (opc-ceo-core)           │
├─────────────────────────────────────────────────────────────┤
│   定位: 规范制定者             定位: 规范执行者              │
│   思考框架: TRS (3 维度)       思考框架: WRAP (4 维度)       │
│                                                              │
│   职责:                       职责:                         │
│   ✅ 制定公司运营标准         ✅ 执行 Coach 制定的规范       │
│   ✅ 评估 CEO 表现            ✅ 日常运营和任务分发          │
│   ✅ 提供成长指导             ✅ 协调各专业角色              │
│   ✅ 识别能力缺口             ✅ 验收任务结果                │
│   ✅ 自我进化                 ✅ 自我进化                    │
│   ✅ 记录到 Memory            ✅ 记录到 Memory               │
│                                                              │
│   不做:                       不做:                         │
│   ❌ 直接执行业务任务         ❌ 制定运营标准                │
│   ❌ 替代 CEO 做决策          ❌ 评估自身表现                │
│   ❌ 越级指挥子角色           ❌ 修改 Coach 规范             │
└─────────────────────────────────────────────────────────────┘
```

**触发规则**：

| 角色 | 需要触发词的场景 | 无需触发词的场景 |
|------|------------------|------------------|
| **Coach** | 日常咨询、业务执行 | 规范制定、评估诊断、成长指导、自我进化 |
| **CEO** | 所有业务执行 | 内部状态汇报、自我进化 |
