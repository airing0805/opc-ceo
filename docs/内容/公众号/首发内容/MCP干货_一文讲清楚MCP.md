## 一句话解释

## 打个比方，秒懂

想象你雇佣了一个超级聪明的秘书（Claude）。

### 没有 MCP 的世界

这个秘书只能：

- 看到你给她的文件
- 听你说的话
- 根据已有知识回答问题

**但她不能：**

- 查实时天气
- 读你电脑里的文件

现在，你给这个秘书配了一套"工具套装"：

- 手机（查天气、搜网页）
- 电脑（读写文件、操作数据库）
- 邮箱（自动发邮件）

**一句话：AI 从"只会聊"变成了"真能干"。**

**现在，只要支持 MCP，所有工具都能通用。**

## MCP 的核心概念

理解 MCP，需要掌握 4 个关键词：

### 1. 服务器（Server）

服务器是提供能力的"服务端"。

比如：

- 文件系统服务器 → 读写文件
- GitHub 服务器 → 操作仓库
- Slack 服务器 → 发送消息

每个服务器专注于某一类功能。

### 2. 客户端（Client）

客户端是使用能力的"消费端"。

比如：

- Claude Desktop
- Claude Code
- Cursor
- Windsurf

### 3. 工具（Tool）

工具是服务器暴露的可执行功能。

例如文件系统服务器提供：

- `read_file` - 读取文件
- `write_file` - 写入文件
- `list_directory` - 列出目录
- `search_files` - 搜索文件

### 4. 资源（Resource）

资源是服务器暴露的可读数据。

通过 URI 标识，比如：

- `file:///path/to/document.txt`
- `github://repo/issue/123`

## MCP vs 传统方案

| 维度 | 传统方案 | MCP 方案 |
|------|---------|---------|
| 集成方式 | 每个数据源单独开发 | 统一协议，一次开发 |
| 可复用性 | 只能用于一个 AI 平台 | 所有 MCP 客户端通用 |
| 安全性 | 各自实现，参差不齐 | 内置权限模型 |
| 维护成本 | 高，每个集成都要维护 | 低，协议统一 |

**MCP 就像 USB 接口统一了外设连接一样，统一了 AI 与外部世界的连接。**

---

## 一个简单的配置示例

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/yourname/Documents"
      ]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "your-token-here"
      }
    }
  }
}
```

配置完成后，Claude 就能：

- 读写你指定的文件夹
- 操作你的 GitHub 仓库
