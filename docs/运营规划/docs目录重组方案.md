## 一、重组目标

### 1.1 核心目标

| 目标 | 描述 | 优先级 |
|------|------|--------|
| 结构清晰化 | 消除职责交叉，明确每个目录的单一职责 | P0 |
| 命名规范化 | 所有目录/文件使用中文命名（除技术文档外） | P0 |
| 归档规范化 | 历史版本归档到统一位置，与活跃内容分离 | P1 |
| 层级扁平化 | 减少不必要的嵌套层级 | P2 |

### 1.2 预期效果

- 根目录零散文件：3 → 0
- 工作日志位置：2 处 → 1 处
- 历史归档位置：分散 → 统一
- 目录层级深度：最大 4 层 → 最大 4 层（保持）

## 三、重组方案

### 3.1 目标结构

```
docs/
├── 规划体系.md              ✅ 保持在根目录（作为总入口索引）
│
├── 战略规划/               # 长期方向（6-12月）
│
├── 版本规划/               # 版本迭代
│   ├── v2-技能规划/        # 当前活跃版本
│   ├── v2-技能规划-已完成/
│   ├── v3-技能规划/
│   └── v4-技能规划/
│
├── 运营规划/               # 日常运营
│   ├── 内容生产规范.md     # 从根目录移入
│   ├── 内容发布规范.md     # 从根目录移入
│   ├── 工作日志/           # 统一的工作日志位置
│   ├── 内容发布/           # 从根目录移入（发布内容+追踪）
│   ├── 内容规划/           # 从根目录移入（选题库+执行规划）
│   └── ...
│
├── 技术规划/               # 技术专项
│
├── 沟通文档/               # 运营沟通记录
│   ├── 工作日志/           # 删除，内容合并到运营规划
│   └── ...
│
├── 内容/                   # 产品内容（电子书、课程）
│
├── 归档/                   # 历史归档
│   ├── v1-技能规划/        # 从版本规划移入
│   └── 24h-round/          # 重命名后移入
│
└── tmp/                    # 临时文档（思考古迹）
```

### 3.2 设计原则

1. **规划体系.md 保留在根目录** - 作为文档总入口和导航索引
2. **内容相关目录统一到运营规划** - 内容发布规范、内容生产规范、内容发布、内容规划
3. **工作日志统一位置** - 保留运营规划/工作日志/，删除沟通文档/工作日志/
4. **历史版本统一归档** - v1-技能规划移至归档目录

## 六、风险评估

### 6.1 风险列表

| 风险 | 等级 | 描述 | 缓解措施 |
|------|------|------|----------|
| 引用断裂 | 中 | 其他文档中可能引用了被移动文件的路径 | 移动后检查并更新 CLAUDE.md、规划体系.md |
| Git 状态混乱 | 低 | 大量文件移动可能导致 Git 历史难以追踪 | 使用 `git mv` 命令保留历史 |
| 用户习惯改变 | 低 | 用户可能习惯原有路径 | 更新 CLAUDE.md 中的文档索引 |
| 临时文件丢失 | 极低 | 移动过程中可能遗漏 | 操作前先列出所有文件 |

### 6.2 回滚方案

如果重组后出现问题，可按以下步骤回滚：

1. 使用 `git checkout .` 恢复所有未提交的更改
2. 使用 `git reset --hard HEAD` 恢复到重组前的状态
3. 重新评估重组方案
